# AI Virtual Church

A fully AI-conducted worship service: intake -> worship -> prayer -> scripture ->
sermon -> testimony -> offering -> closing hymn -> benediction. Every spoken segment
is generated from the logged-in user's own input, so no two services are identical.

## Music: two interchangeable sources
Each user picks one (stored on `users.music_source`, locked per session):
- **suno**    — original worship music generated by AI, stored in object storage.
- **youtube** — an existing worship track, embedded via the YouTube player (no download).

The choice is a strategy (`workers/strategies/`); both return the same `MusicResult`,
so the rest of the pipeline is source-agnostic.

## Run (dev)
Backend:
    cd backend && composer install && php artisan migrate && php artisan serve

Queues (Laravel side):
    php artisan queue:work

Workers (Python side):
    cd workers && pip install -r requirements.txt
    python bridge.py
    celery -A tasks.celery_app worker -Q ai:sermon,ai:music,ai:avatar -c 4

Frontend:
    cd frontend && npm install && npm run dev

## Flow
1. Vue intake form -> POST /service/start, then /service/{token}/intake
2. Crisis intercept runs BEFORE the queue; flagged intakes get static resources, no LLM.
3. DispatchServiceJob publishes JSON to Redis list `ai:intake`.
4. bridge.py BLPOPs it -> Celery `orchestrate` -> fans out to sermon + music tasks.
5. Workers POST finished assets to /api/internal/asset-ready (shared-secret).
6. Laravel stores the asset and pushes it to the client over WebSockets.
